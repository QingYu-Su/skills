# AI 自动化工作流开发 Skill

## 核心理念：AI 助手中台

本 Skill 围绕一个核心理念：构建一个 **AI 助手中台**，它作为用户与各种工具/应用之间的中间层。

### 架构模型

```
用户 ←→ [AI 助手中台 (n8n)] ←→ 各种应用和工具
                ↑
        企业微信 / 钉钉 / 飞书 / Telegram / ...
```

### 中台的两个核心能力

**1. 信息汇聚与转发（入站）**
- 各个应用（GitHub、监控告警、日历提醒、邮件通知等）通过 Webhook 将信息推送到中台
- 中台对信息进行加工处理（摘要、格式化、优先级判断、AI 摘要等）
- 将处理后的信息通过用户偏好的渠道（企业微信/钉钉/Telegram 等）转发给用户

**2. 指令接收与执行（出站）**
- 用户通过任意接入的渠道发送消息/指令给中台
- 中台解析指令意图，执行相应操作（调用 API、触发工作流、查询信息等）
- 将执行结果返回给用户

### 典型工作流模式

#### 模式一：外部事件通知 → 处理 → 推送给用户

```
[外部 Webhook 触发器] → [数据加工/AI 处理] → [消息推送节点] → 用户
```

场景示例：GitHub 有新 Issue → AI 总结 → 通过企业微信推送给用户

#### 模式二：用户发消息 → 中台处理 → 执行动作

```
[消息接收触发器] → [意图识别/指令解析] → [执行对应操作] → [返回结果]
```

场景示例：用户在企业微信发送"今天有什么会议" → 中台查询日历 → 返回会议列表

#### 模式三：跨应用桥接

```
[应用A 消息接收] → [格式转换/内容处理] → [应用B 消息发送]
```

场景示例：企业微信收到消息 → 附加处理后转发给指定用户

#### 模式四：AI 增强处理

```
[消息接收] → [AI 节点(总结/翻译/分析)] → [结果推送]
```

场景示例：收到长文通知 → AI 提取要点 → 推送摘要给用户

## 开发前置检查：n8n MCP 连通性

每次开始开发前，必须先验证 n8n MCP 连接是否正常。详见 `references/n8n-development.md` 中的 MCP 配置教程。

- 调用 `n8n_health_check` 验证连通性
- 如果失败，引导用户按教程配置 MCP

## 技术底座：n8n

n8n 是本工作流的技术底座，负责串联各个节点实现自动化。n8n 的核心概念、MCP 工具使用、工作流开发流程等详见 `references/n8n-development.md`。

### n8n 在中台架构中的角色

- **工作流引擎**：通过可视化节点编排实现自动化逻辑
- **集成枢纽**：通过社区节点连接各种第三方服务
- **数据处理管道**：内置的 Set、Code、IF 等节点实现数据转换和条件路由
- **Webhook 服务**：作为消息入口接收各应用的推送

当用户提到某个具体渠道时，读取对应的参考文档获取节点配置、凭证、开发模式等详细信息。

### Webhook 路径管理

每个接入渠道的消息接收都通过 Webhook 实现。关键原则：
- 每个应用（如企业微信的每个 Agent）有唯一的消息接收 URL
- 多个工作流可共享同一个 Webhook 路径（同一应用）
- 不同应用使用不同的路径和凭证
- 具体路径值属于敏感信息，不写入 Skill，开发时由用户提供
